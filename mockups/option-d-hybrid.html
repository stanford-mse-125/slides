Option D: Hybrid Workflow (PowerPoint + Jupyter)

Approach

Use PowerPoint for visual presentation slides and a Jupyter notebook for live coding demos. The instructor switches between the two applications during lecture.

  • PowerPoint handles: opening hooks, “Who am I?”, visual storytelling, discussion prompts, diagrams, course structure – anything that benefits from polished layout, images, or the Stanford visual identity.
  • Jupyter notebook handles: loading data, making plots, answering student questions with code – anything interactive or exploratory.
  • Switching pattern: PPT, Jupyter, PPT, Jupyter, roughly alternating every 10-15 minutes. The instructor keeps both apps open and uses Cmd-Tab (or a second monitor) to switch.

Lecture 1: Slide-by-Slide Plan (80 min)

Time Tool Segment Content
0-5 PPT Slides 1-2 Title slide. “Who am I?” – background, research, why data matters in MSE.
5-10 PPT Slides 3-4 Opening hook: the $500K Medicare fine for high ERR. Discussion prompt: “What would you check first?”
10-20 Jupyter Demo 1 Load CMS hospital data. Compute excess readmission ratios. Make a histogram of ERR across hospitals. Students see pandas, matplotlib for the first time. (~10 min)
20-28 PPT Slides 5-7 Debrief: what we learned from the data. Three acts of the course (description, prediction, causal inference). Course structure and logistics.
28-40 Jupyter Demo 2 Load Airbnb listings data. Explore with .head(), .describe(), .value_counts(). Make a price distribution plot. Ask: “What’s weird about this distribution?” (~12 min)
40-48 PPT Slides 8-10 Missing data motivation – why .describe() might lie to you. Discussion: “When is missing data not random?” Transition to the idea that data cleaning is not optional.
48-55 Jupyter Demo 3 Show missing data patterns in the Airbnb data: .isna().sum(), a missing-data heatmap. Quick live Q&A – students suggest columns to check, instructor types it live. (~7 min)
55-60 PPT Slide 11 Key takeaways: data is messy, summary statistics can mislead, visualization reveals what numbers hide.
60-65 PPT Slide 12 Next time preview (probability + distributions). Feedback: one-minute paper or poll.

Total: ~65 min of content, leaving buffer for transitions, questions, and a break if needed.

Pros

  • Best visual quality for presentation slides. PowerPoint (or Keynote) is purpose-built for polished layouts, images, and branding. The Stanford theme looks right without CSS hacking.
  • Live interactivity for code. Jupyter lets the instructor respond to student questions by coding answers on the spot, not just clicking to the next pre-built slide.
  • Students see the real tool. They will use Jupyter for homework. Seeing the instructor work in it normalizes the environment and demonstrates good workflow habits (restart-and-run-all, markdown cells for notes, etc.).
  • Each tool does what it is best at. No compromises: no ugly code-on-slides, no awkward “presentation mode” in a notebook.

Cons

  • Two-tool workflow. Switching between apps during lecture adds friction and a small risk of fumbling (wrong window, lost cursor, notebook kernel dies while in PPT).
  • PowerPoint files are hard to version-control. Binary .pptx files produce meaningless diffs in git.
  • Cannot auto-generate slides from book content as easily. A pure-text pipeline (Quarto, RISE) can pull prose and code from the book source; PowerPoint cannot.
  • Two artifacts per lecture. Each lecture produces both a .pptx and a .ipynb, which must be kept in sync, distributed to students, and updated together.

Mitigations

Version control for PowerPoint

Use python-pptx to generate slides programmatically from a Python script. Version-control the script (plain text, clean diffs), not the .pptx binary. The script can pull content from the book’s markdown source, so updates propagate automatically.

# Example: generate Lecture 1 slides
python generate_slides.py --lecture 1 --template stanford_theme.pptx --output lec01.pptx

Version control for notebooks

Jupyter notebooks are .ipynb (JSON). They version-control adequately with nbstripout to remove outputs before committing, plus jupytext to maintain a paired .py or .md file for readable diffs.

Alternative: Quarto revealjs for everything

Instead of maintaining two tools, use Quarto with revealjs output for the entire lecture:

  • The Stanford theme can be applied as a custom SCSS theme for revealjs.
  • Presentation slides (text, images, diagrams) are written in markdown with full layout control.
  • Live coding portions use executable Python cells embedded in the same document.
  • One source file (.qmd) produces both the slide deck and the runnable code.
  • Trade-off: visual polish for purely presentational slides may require more CSS work than PowerPoint, but the single-source workflow eliminates the sync problem entirely.

This alternative is worth prototyping. If the revealjs output looks good enough with the Stanford theme, it dominates the hybrid approach on workflow simplicity while retaining all the interactivity benefits.
